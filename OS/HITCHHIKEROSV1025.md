════════════════════════════════════════
 
Purpose
Enable structured narrative branching and CYOA-style agency tracking for interactive fiction projects.
Project tool, not global OS rule.
 
I. FORK DEFINITION
A fork exists when:
• a character decision is made
• that decision changes risk, resources, or relationships
• world state changes as a result
No lasting consequence → not a fork.
 
II. STANDARD FORK TYPES
 
LOGISTICAL
Resource tradeoffs (battery, fuel, light, food)
 
POSITIONAL
Movement/location (stay/move/scout/timing)
 
RELATIONAL
Trust/social dynamics (side/protect/conserve)
 
MORAL
Value-revealing (self vs others, comfort vs vigilance)
 
III. NODE RECORD FORMAT
 
NODE ID:
Phase:
Location:
Context:

Immediate Cost:
Future Risk:

Immediate Cost:
Future Risk:

IV. BRANCH TAG SCHEMA
Format:
p#.location.event
 
Examples:
p2.library.mirrorloss
p3.library.fuelsplit
p4.road.departure
 
V. FORK RHYTHM RULE
Introduce forks when:
• margins shrink
• stakes change
• decisions are irreversible
• every 2–3 meaningful beats
 
Avoid:
• constant forks
• menu prompts
• fake choices
 
VI. AGENCY CHECK (COMPACT)
A valid fork must:
1) present a real decision
2) impose cost on each option
3) reveal values
4) change the world state
5) preserve grounded tone
 
Golden Rule:
Agency = choosing which loss to accept.
 
VII. STORAGE RULE
Forks are logged in a branch index,
not embedded in story prose.
Referenced by Node ID when needed.
